# 实习日记 — 数据中心监控大屏开发

## 2026年7月8日 星期三 晴

今天是大数据专题实训的第三天，任务是基于服务器监控数据，搭建一个数据中心运行监控的可视化大屏。全天分为理论授课和项目实战两部分，从大数据概念到可视化大屏落地，走了一个完整的闭环。

### 上午：理论授课 + 数据准备

上午先进行了三场直播授课。

**第一场**介绍了大数据的行业背景和技术演进。2013年被称为"大数据元年"，技术栈从早期的数据仓库、Hadoop分布式计算，发展到Spark内存计算和Flink实时处理。大数据具有4V特征：数据量大（Volume）、类型繁多（Variety）、价值密度低但整体价值高（Value）、速度快时效高（Velocity）。讲师重点讲解了Hadoop生态系统，包括HDFS分布式存储（多副本机制）、MapReduce计算模型（分治与聚合思想）、Hive（SQL接口离线分析）和HBase（列式存储海量查询）等组件，以及Protobuf技术在日志结构化存储中的优势。

**第二场**围绕数据分析工具和应用场景展开。Python凭借Pandas、NumPy和Spark框架已成为大数据和AI领域的主流工具，远超R语言。数据可视化强调清晰有效地传达信息，而非单纯追求炫酷——讲师以供暖工厂从Excel表单到可视化大屏的转变为例，展示了九宫格布局和关键指标趋势图如何提升管理效率。应用场景方面，分享了浙江省充电桩监管平台、联通高价值用户网络体验优化、保险行业精准营销等实际案例。技术组件上对比了Spark（内存计算+MLlib机器学习库）的高效性、Kafka作为消息中间件解决高并发数据缓冲，以及Flink支持批流一体、通过窗口机制实现实时与历史数据关联分析。

**第三场**聚焦数据采集、质量治理和数仓建模。数据类型分为结构化（数据库表）、半结构化（JSON、日志）和非结构化（音视频，占比超80%）。数据清洗是耗时耗力的"体力活"，常见问题包括缺失值、冗余数据、爬虫干扰和格式不统一，需要保障唯一性、及时性、准确性、有效性、完整性和一致性六个质量维度。数仓建模方面，对比了传统E-R图和企业级多维模型（事实表+维度表），展示了从早期ETL架构到大数据时代含Kafka、Spark/Flink的架构演进。

理论课后进入实操环节。

实训数据是四份 `.dat` 格式的TSV文件，分别记录了服务器基础信息（host_detail，20台）、监控指标定义（mod_detail，55个）、磁盘监控时序数据（disk_tsar，12,000条）和性能监控时序数据（pref_tsar，67,200条）。

首先用 Docker 启动 MySQL 8.0 容器，将四份数据导入 monitor 库。MySQL 8 默认的 `caching_sha2_password` 认证插件与 Navicat 不兼容，需要切换到 `mysql_native_password`。导入时还踩了一个坑——`--secure-file-priv` 限制了 LOAD DATA 的文件路径，必须把文件拷贝到 `/var/lib/mysql-files/` 才能导入。

导入后分析表关系，画出了ER图：host_detail 和 mod_detail 是维度表，disk_tsar 和 pref_tsar 是事实表，通过 hostid 和 mod 字段关联，构成典型的星型模型。外键引用完整性验证全部通过，零孤儿数据。

课程后半段学习了时间戳解析——数据里的 `ts` 是毫秒级 Unix 时间戳，用 `FROM_UNIXTIME(ts/1000)` 转换成可读时间。还做了按小时汇总的练习，用 `GROUP BY` 配合 CASE WHEN 透视出 CPU、内存、负载等指标的小时级均值。

### 下午：大屏开发实战

下午的任务是把分析结果变成可视化的监控大屏。

**技术选型**上，对比了多种方案后选择了 **纯HTML + ECharts + Flask** 的组合。理由是：单页大屏不需要 Vue/React 框架；ECharts 是国内大屏标配，暗色主题和动效开箱即用；Flask 写 SQL 查 MySQL 只需80行代码。项目取名 datacenter-screen，Git 仓库推到了 GitHub。

**后端**写了10个 API 接口：基础统计、机房健康度、主机风险排名、磁盘使用率TOP5、CPU趋势、负载趋势、磁盘读写汇总、内存TOP5、网络流量、告警列表。每个接口对应一条聚合 SQL，pymysql 直连 Docker 里的 MySQL。

**前端**是重头戏。用 CSS Grid 搭建了响应式布局，顶部标题栏+统计卡片，中间图表区两列并排，只有 CPU 趋势独占一行（最重要指标）。ECharts 配置了统一的暗色主题：深蓝底色 `#0a0e27`，青蓝主色 `#00d4ff`，绿色正常、橙色警告、红色严重。图表类型覆盖了雷达图、折线图、面积图、堆叠柱状图、水平柱状图和双Y轴折线图。

**遇到的最大坑**是 ECharts 图表全部空白。排查了半天，写了好几个诊断页面，最终定位到：图表容器在页面布局完成前初始化（0×0 尺寸），导致 canvas 无法渲染。解决方案是在每个渲染函数中同步执行 `dispose → init → setOption`，确保容器已有尺寸时才创建图表实例。然而这个问题到下班前仍未完全解决——测试页（硬编码数据）能正常显示图表，但大屏页面（API异步数据）的图表模块仍然空白。这是明天需要攻克的首要问题。

**布局**也经过多轮迭代：最开始是固定 `100vh` 的三列大屏，太拥挤；改为可滚动，又觉得信息密度不够；最后在"全宽独占"和"两列并排"之间找到了平衡——只有 CPU 趋势全宽，其余8个模块分4行两列展示。页面 `max-width: 1500px` 居中，避免在超宽屏上过于松散。

**细节打磨**：统计卡片加了渐变发光顶边 + 逐个淡入动画，告警列表有闪烁红点，背景用 Canvas 画了60个缓慢上浮的青色粒子 + 近距离连线（模拟网络拓扑感）。

前端代码按模块拆分为 `api.js`（数据层）、`charts.js`（图表渲染）、`app.js`（主控逻辑）和 `particles.js`（背景粒子），CSS 也拆成 `theme.css`、`layout.css`、`components.css` 三件套，方便后续维护。

### 收获与反思

1. **ER建模**是数据分析的基础，表关系理清了后面写 SQL 才不会出错
2. **时间戳处理**和**窗口聚合**是时序数据的核心操作，FROM_UNIXTIME + GROUP BY + CASE WHEN 是标配
3. **ECharts 大屏**的难点不在图表配置，而在**容器尺寸管理**——Flex/Grid 里嵌套 ECharts 必须保证 init 时容器有非零尺寸
4. **模块化**能显著降低心智负担，100行的 index.html + 独立 JS/CSS 文件比之前 400 行的单文件好维护太多
5. **调试方法论**：遇到诡异的渲染问题，先写最小复现页面（test.html → test2.html → test3.html），排除法定位，比盲目改代码高效得多

## 摘要

今日实训涵盖理论授课与项目实战。理论部分学习了大数据4V特征、Hadoop/Spark/Flink技术演进、Python数据分析工具链、数据质量六维度治理、以及事实表与维度表的多维建模方法。实操部分将四份服务器监控TSV数据导入Docker MySQL，绘制星型ER模型，练习毫秒时间戳转换和小时级聚合统计；随后选用Flask+ECharts+纯HTML技术栈，编写10个RESTful API，搭建含雷达图、折线图、面积图等7类图表的暗色监控大屏。踩坑ECharts异步渲染时容器零尺寸导致空白，通过编写最小复现测试页逐层排查。代码实现JS/CSS模块化拆分，Git控版推送GitHub。最大收获：理解了大数据从采集、清洗、建模到可视化的完整链路，以及前端图表渲染对容器尺寸的强依赖。
